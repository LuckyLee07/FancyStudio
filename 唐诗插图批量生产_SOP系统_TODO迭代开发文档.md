## 1. 产品重新定标

### 1.1 一句话定位

本产品是一个“唐诗插图生产操作系统”：将 300 首唐诗从结构化内容，经过 AI 策划、美术定向、批量生成、自动质检、人工审核和返工收敛，转化为可追溯、可交付、风格一致的插图资产。

### 1.2 它不是什么

- 不是面向普通用户的一次一图生成器；
- 不是以作品浏览为核心的画廊；
- 不是只负责拼 Prompt 的小工具；
- 不是全自动、无人负责的“抽卡系统”；
- 不是首版就覆盖复杂组织权限和客户协同的 SaaS。

### 1.3 核心成功指标

**北极星指标：每周通过终审并可直接交付的诗词插图数量。**

| 指标 | Alpha 目标 | Beta 目标 | 300 首生产目标 |
|---|---:|---:|---:|
| 单周终审成品 | ≥ 10 首 | ≥ 30 首 | ≥ 50 首 |
| 单首人工操作时长 | ≤ 30 分钟 | ≤ 20 分钟 | ≤ 15 分钟 |
| 首轮方向通过率 | ≥ 50% | ≥ 65% | ≥ 75% |
| 生成任务成功率 | ≥ 90% | ≥ 95% | ≥ 98% |
| 自动质检召回明显废图 | ≥ 70% | ≥ 80% | ≥ 85% |
| 平均每首进入人审候选 | ≤ 12 张 | ≤ 8 张 | ≤ 6 张 |
| 成品链路可追溯率 | 100% | 100% | 100% |

> 指标基线必须通过首批 20 首试生产校准；在没有真实生产数据前，不将上述数字用于绩效承诺。

### 1.4 首版范围

首版聚焦单团队、单项目空间、桌面端后台：

- 支持导入并管理 300 首诗词数据；
- 先以 20 首基准诗跑通生产闭环，再扩展到 100 首和 300 首；
- 支持 6 个可版本化风格包；
- 支持 AI 需求生成、三方向策划、批量排产、生成、质检、审核、返工和导出；
- 所有中间产物、人工决策和最终资产可追溯；
- 首版不做复杂 RBAC，但所有操作必须记录操作者字段，为后续多人协作预留。

## 2. 当前实现审计与重构边界

### 2.1 可复用能力

| 当前能力 | 处理方式 | 说明 |
|---|---|---|
| Python 本地服务和 API 骨架 | 保留并分层 | 可继续作为单机版运行时 |
| OpenAI 图像生成 / 编辑适配 | 保留并封装 | 移入 Provider Adapter，不与业务状态耦合 |
| Demo SVG 生成 | 保留 | 用于无 API Key 演示和自动化测试 |
| Project / Job / Image 基础对象 | 数据迁移后复用概念 | 字段不足，需要替换成生产对象 |
| explore / converge 两类生成 | 保留为任务意图 | 分别映射“探索新方向”和“基于候选返工” |
| candidate / selected / rejected / final 决策 | 扩展 | 加入自动质检、待复审、阻塞和终审状态 |
| 图片父子衍生关系 | 保留 | 作为返工谱系和版本追溯基础 |
| 反馈标签和基础 QC 字段 | 扩展 | 改为结构化 ReviewDecision 与 QCResult |
| 响应式页面壳与视觉语言 | 有选择保留 | 后台工作台优先信息密度，不延续消费型首页结构 |

### 2.2 必须重构

| 问题 | 影响 | P0 改造 |
|---|---|---|
| 以“项目 + 单次生成”为核心 | 无法表达批量生产阶段 | 改为“生产项目 + 诗词条目 + 阶段产物” |
| 诗词仅为静态样例 | 无法管理 300 首状态 | 增加导入、校验、版本和批量状态 |
| `state.json` 单文件持久化 | 并发、迁移、查询和恢复风险高 | 使用 Python `sqlite3` 建立关系数据层 |
| Job 只有简单运行状态 | 无依赖、暂停、恢复、批量控制 | 引入 Batch、Task、Attempt 和状态机 |
| 需求与画面方向只存在于 Prompt | 无法审核、复用、锁字段 | 独立 RequirementCard、DirectionProposal |
| 风格只是生成参数 | 无法版本管理和跨诗一致性检查 | 独立 StylePackVersion 与基准测试 |
| QC 主要依赖人工标签 | 人审负担大且口径不统一 | 引入技术 QC、语义 QC、历史风险和去重 |
| 无成品导出记录 | 无法正式交付 | 引入 FinalAsset、ExportPackage、Manifest |
| 无审计与成本账本 | 无法定位变更和控制预算 | 引入 AuditEvent、UsageRecord、BudgetGate |
| 页面以单诗编辑和画廊为主 | 生产人员频繁跳转 | 改为看板、批量操作、异常优先的 SOP 后台 |

## 4. 目标 SOP 与阶段门禁

### 4.1 全链路

~~~mermaid
flowchart LR
    A["项目创建与诗词导入"] --> B["内容校验"]
    B --> C["AI 需求生成"]
    C --> D{"内容编辑通过?"}
    D -- "否" --> C
    D -- "是" --> E["生成 3 个画面方向"]
    E --> F{"美术指导通过?"}
    F -- "否" --> E
    F -- "是" --> G["批次排产与预算闸门"]
    G --> H["图像生成"]
    H --> I["自动质检与去重"]
    I --> J{"可进入人审?"}
    J -- "否" --> K["自动淘汰或异常处理"]
    K --> G
    J -- "是" --> L["候选审片"]
    L --> M{"终审通过?"}
    M -- "返工" --> N["锁定保持项与变化项"]
    N --> G
    M -- "通过" --> O["成品锁定、导出与归档"]
~~~

### 4.2 阶段定义

| 阶段 | 责任角色 | 输入 | 系统产物 | 退出门禁 | 失败去向 |
|---|---|---|---|---|---|
| S0 项目建档 | PM / CE | 交付规格、诗词文件 | ProductionProject、PoemRecord | 数据完整率 100% | 待补数据 |
| S1 内容校验 | CE | 原文、注释、来源 | ContentVersion | 正文与来源已确认 | 内容冲突 |
| S2 需求策划 | AI / CE | ContentVersion、全局指令 | RequirementCard | 必须 / 禁止 / 风险已确认 | 退回 AI 重写 |
| S3 美术定向 | AI / AD | RequirementCard、风格包 | DirectionProposal × 3 | 至少 1 个方向批准 | 退回补方向 |
| S4 批次排产 | OP | 已批准方向、规格、预算 | GenerationBatch、Task | 预算与并发检查通过 | BudgetBlocked |
| S5 生成质检 | AI / 系统 | Task、PromptVersion | CandidateImage、QCResult | 通过硬门槛且非重复 | 淘汰 / 异常 |
| S6 人工审片 | AD / CE | 候选、QC 和上下文 | ReviewDecision | 入选、返工或淘汰明确 | 返工任务 |
| S7 终审交付 | PM / AD | 入选图、排版规格 | FinalAsset、ExportPackage | 资产清单完整且终审签字 | 退回审片 |

### 4.3 阶段通用规则

- [x] P0 每次阶段推进都必须有明确输入、输出、操作者和时间；
- [x] P0 未通过上游门禁的诗词，不允许静默进入下游批量任务；
- [x] P0 批量操作先展示影响范围，再执行，并返回逐条结果；
- [x] P0 所有自动产物均可人工编辑，编辑后产生新版本而非覆盖历史；
- [ ] P0 阻塞项必须记录 `blocked_reason`、责任角色和建议动作；
- [x] P0 服务重启后恢复到最后一次已持久化状态，不重复创建生成任务；
- [ ] P1 支持阶段 SLA、逾期提醒和责任人筛选；
- [ ] P1 支持跨阶段批注和 @责任人。

9. 美术指导用快捷键完成候选审片；
10. 从候选创建返工，生成后能追溯到父图；
11. 锁定终审成品，导出纯图和 Manifest；
12. 从导出资产反查完整生产链路。

### 13.3 发布级别

| 级别 | 数据规模 | 允许用户 | 发布门禁 |
|---|---:|---|---|
| Prototype | 3 首 | 产品 / 研发 | SOP 与状态走通 |
| Alpha | 20 首 | 内部生产团队 | 端到端闭环、60 张稳定批次、可导出 |
| Beta | 100 首 | 完整内容与美术团队 | 连续 2 周生产、指标稳定、无研发改库依赖 |
| Production | 300 首 | 正式项目团队 | 备份恢复、成本可控、7 天稳定运行、全部可追溯 |

### 13.4 一票否决项

以下任一问题未解决，不得升级发布级别：

- 出现重复计费或无法解释的任务重复；
- 成品无法追溯到需求、方向、Prompt 和模型版本；
- 服务重启造成任务或人工审核结果丢失；
- 未通过方向审批仍可批量生成；
- 预算硬上限失效；
- 导出包漏图、覆盖旧版本或 Manifest 不一致；
- 自动质检故障却将图片标记为通过；
- 数据只能由研发直接改文件或数据库才能恢复。

## 16. 总体 Definition of Done

一个功能只有同时满足以下条件才算完成：

- [ ] 有明确用户入口、空状态、加载状态、成功状态和错误状态；
- [ ] 有状态转换和数据持久化，不依赖页面内临时状态；
- [ ] 有权限 / 责任角色和审计事件；
- [ ] 有批量场景和部分失败处理；
- [ ] 有单元或集成测试，关键路径有端到端验证；
- [ ] 有可观测字段，出现问题能定位到对象和请求；
- [ ] 有验收样例，PM、实际操作者和 QA 均已走查；
- [ ] 无需研发直接改数据才能完成正常操作；
- [ ] 文档、字段、页面术语和 API 命名一致；
- [ ] 未引入超出当前迭代范围的隐性依赖。

一首诗只有同时满足以下条件才算“生产完成”：

- [ ] 内容版本已审核；
- [ ] Requirement 已批准；
- [ ] 至少一个 Direction 已批准；
- [ ] 生成任务和 Prompt 可追溯；
- [ ] 自动 QC 已运行或有人工替代记录；
- [ ] 美术和内容终审均有结论；
- [ ] FinalAsset 文件、规格和校验和有效；
- [ ] 已进入 ExportPackage 并有 Manifest；
- [ ] 无未关闭的高风险历史或版权问题。

## 20. 最终交付判断

> 一个内容生产团队可以在不依赖研发改数据、不依赖个人记忆、成本可控且全程可追溯的前提下，持续把 300 首唐诗推进为风格统一、内容准确、通过终审并可直接交付的插图资产。

任何不能提高批量吞吐、质量稳定性、决策效率、恢复能力或交付可追溯性的功能，都不应抢占 P0 排期。

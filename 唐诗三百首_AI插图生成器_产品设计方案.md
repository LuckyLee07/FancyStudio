# 《唐诗三百首 AI 插图生成器》产品设计方案

**文档版本：** V1.0  
**文档状态：** 产品立项与 MVP 设计稿  
**更新日期：** 2026-07-16  
**建议产品名：** 唐诗绘卷（暂定）  
**一句话定位：** 面向古诗词内容创作者与教育团队的 AI 插图生产工作台，让系统持续生成、自动筛选并根据人工审美反馈迭代，用户每天只需挑选可用作品。

> 核心判断：本产品不是“输入一首诗、等待一张图”的生成按钮，而是“结构化诗词资产 + 风格资产 + 自动生产流水线 + 人工审美反馈回路”。产品壁垒来自整套作品的一致性、可持续生产效率、历史文化约束和用户偏好数据，而非单一模型。

## 1. 执行摘要

### 1.1 产品机会

唐诗插图创作的主要困难并不是“能不能生成图片”，而是：如何连续生成足够多、彼此风格一致、与诗意相关、历史穿帮较少且可以直接用于教育或出版场景的作品。通用聊天产品可以生成单张插图，但无法天然承担诗库管理、批量排产、版本追踪、成本治理、自动筛选、整套美术一致性与偏好积累。

本产品将唐诗插图生产拆成可追踪、可复用、可迭代的流水线：

```text
诗词库 → 诗意解析 → 画面策划 → 风格匹配 → Prompt 编译
       → 预算检查 → 批量出图 → 自动质检 → 候选池
       → 人工反馈 → 偏好更新 → 变体任务 → 成品归档
```

### 1.2 推荐切入点

第一阶段不直接面向大众消费者，而应先做成单人或小团队使用的“唐诗插图生产工作台”。首要用户是产品发起人本人、教育内容团队和古诗词内容创作者。先验证每日自动生产和筛选闭环，再扩展为面向普通用户的轻量 App。

### 1.3 MVP 目标

在 8-10 周的小团队全职开发周期，或 4-6 个月的个人业余开发周期内，完成以下闭环：

1. 内置 30 首高频唐诗和 6 个原创风格包；
2. 每首诗可生成 3 个结构化画面方案；
3. 自动编译 Prompt、创建任务并调用一个可替换的图像后端；
4. 每日定时生成，自动质检后形成 20-40 张候选图；
5. 用户可收藏、淘汰、标记优点并发起衍生；
6. 系统按反馈调整下一轮风格与构图权重；
7. 所有任务受每日张数、预算、并发和失败重试上限控制。

### 1.4 成功定义

MVP 成功不以“总共生成多少张”为标准，而以“每天无需等待即可看到值得挑选的候选图，并且单位可用图成本持续下降”为标准。建议北极星指标为：

> 每周进入成品库的人工确认可用插图数。

## 2. 产品定位与目标

### 2.1 产品定位

**产品类别：** AI 原生内容生产工具 / 古诗词视觉资产工作台。  
**核心产物：** 无文字插图、诗词海报、儿童学习卡、课程配图、绘本页和短视频画面素材。  
**核心交互：** 系统持续生产，用户集中筛选；用户不是等待者，而是美术总监。  
**产品形态：** Web 管理工作台为主，未来增加轻量移动端和 ChatGPT 自然语言控制入口。

### 2.2 产品愿景

建立一套可持续生成整套古诗词视觉作品的智能生产基础设施，使个人创作者也能获得接近小型插画工作室的策划、试稿、筛选、迭代与资产管理能力。

### 2.3 阶段目标

| 阶段 | 目标 | 关键交付 |
|---|---|---|
| 原型验证 | 证明“每天自动出候选、第二天直接挑选”可行 | 30 首诗、6 个风格、单机任务库、候选画廊 |
| MVP | 打通自动调度、质检、反馈和二次衍生 | Web 工作台、可替换图像适配器、成本闸门 |
| 产品化 | 支持多人、批量项目、模板导出和稳定运维 | 团队空间、权限、对象存储、运营统计 |
| 平台化 | 扩展至宋词、古文、教材与出版社项目 | 内容包、风格市场、开放 API / SDK |

### 2.4 非目标

- 不把产品做成通用 AI 绘图平台；
- 不承诺历史画面绝对准确，系统提供“历史合理性检查”而非权威考据结论；
- 不在 MVP 中训练专属大模型或复杂推荐模型；
- 不依赖自动操作网页实现高频生产；
- 不把模型生成的中文文字直接作为成品排版；
- 不默认使用具体在世艺术家或知名影视、动漫 IP 的风格名称。

## 3. 目标用户与核心需求

### 3.1 核心用户

#### 用户 A：个人内容创作者 / 产品发起人

- 目标：为唐诗三百首建立一套统一、高质量的插图资产；
- 痛点：每次临时想 Prompt、等待生成、结果难归档、不同诗之间风格不统一；
- 期望：系统夜间自动试稿，白天只需用 15-30 分钟筛选。

#### 用户 B：教育内容团队

- 目标：为 App、课程、练习卡、短视频批量制作配图；
- 痛点：素材量大、规格多、版本多，编辑和美术之间沟通成本高；
- 期望：诗词、画面方案、插图、排版模板和导出规格统一管理。

#### 用户 C：出版社 / 文化内容机构

- 目标：低成本完成前期试稿、风格验证和选题样册；
- 痛点：整套作品的一致性与合规审查要求高；
- 期望：风格版本可追溯、每张图有提示词和审核记录，最终由编辑和美术把关。

### 3.2 次级用户

- 语文教师：制作课堂卡片、课件插图和作业素材；
- 家长与学生：选择诗歌和风格生成学习卡；
- 设计师与插画师：把 AI 作为构图草图和风格研究工具；
- 新媒体运营：生成节气、节日和古诗主题内容。

### 3.3 核心待办任务（Jobs to Be Done）

1. 当我要为一组唐诗制作插图时，我希望系统自动提出多种画面方向，以免每首诗都从空白开始；
2. 当我没有时间等待生成时，我希望系统预先跑出候选图，让我集中挑选；
3. 当我找到喜欢的风格时，我希望系统能将该风格稳定扩展到更多诗；
4. 当候选图数量很大时，我希望系统先剔除明显穿帮、乱码、偏题和低质量图；
5. 当我给出喜欢或不喜欢的反馈时，我希望下一轮能明显更接近我的审美；
6. 当我要交付课程或出版素材时，我希望可以追溯诗词来源、风格版本、生成参数与审核结果。

## 4. 核心价值与使用场景

### 4.1 核心价值

| 价值 | 用户收益 | 产品机制 |
|---|---|---|
| 无需等待 | 打开工作台即可筛选结果 | 24 小时分批调度、每日候选池 |
| 懂诗意 | 避免只做字面图解 | 诗意解析、意象与情绪结构化 |
| 可控一致 | 形成整套视觉语言 | 风格包、人物设定、版本化 Prompt |
| 降低废图 | 减少人工浏览负担 | 自动质检、去重、候选重排 |
| 越用越准 | 高命中风格获得更多资源 | 反馈标签、权重更新、探索保留 |
| 可追溯 | 可复盘、可重现、可审计 | 任务链路、版本号、日志与成本账本 |
| 可交付 | 直接进入教育和内容流程 | 模板排版、规格导出、资产归档 |

### 4.2 典型场景

#### 场景 1：每日候选筛选

系统凌晨为 10 首诗生成探索图，自动质检和去重；早上用户打开“今日候选”，先看推荐前 20，使用快捷键完成收藏、淘汰和继续衍生。

#### 场景 2：整套风格建立

用户在《静夜思》《江雪》《山居秋暝》等代表诗上测试 6 个风格包，选出两种高命中风格；系统随后将其扩展到同类诗，并监控跨诗一致性。

#### 场景 3：单诗深度打磨

某首诗已有满意方向，用户锁定风格和人物设定，只让系统改变镜头、动作、光线或留白，逐轮收敛到成品。

#### 场景 4：教育素材导出

编辑选择一张插图，再选择“竖版古典诗卡”“儿童学习卡”或“16:9 课程配图”模板；系统在生成图之外完成诗名、作者、正文、拼音与译文排版。

#### 场景 5：风格实验

用户创建一个不引用具体艺术家名称的风格包，选择 5 首基准诗生成小样，系统统计风格匹配率、收藏率、偏题率和单位可用图成本，达到门槛后再投入生产。

## 5. 产品原则与范围边界

### 5.1 产品原则

1. **先结构化，再生成。** 诗词解析、画面策划和风格描述先形成结构化中间产物；
2. **流水线由代码控制，模型只完成智能节点。** 状态、预算、重试和优先级不交给模型自由决定；
3. **插图与排版分离。** 模型生成无文字画面，程序负责中文排版；
4. **可回放。** 每张图必须能追溯到诗词版本、策划、风格、Prompt、模型与参数；
5. **反馈要轻。** 单张图的主判断不超过 3 秒，细标签只在精选图上补充；
6. **保留探索。** 偏好学习不能导致审美快速收窄；
7. **模型可替换。** 业务层不绑定单一云端或本地图像模型；
8. **人工最终负责。** 自动质检只做辅助筛选，不替代编辑、历史顾问或美术审核。

### 5.2 生产边界

- “24 小时运行”定义为任务队列持续补充与分批执行，不是一个永不退出的会话或死循环；
- 无 API 时，Codex 可负责规划、脚本执行、调度和报告，但仍需本地图像生成器或人工出图环节；
- 自动操作网页仅用于低频原型，不作为正式生产依赖；
- 用户选择“已完成”的诗默认停止探索，只允许手动恢复或生成修订版。

## 6. 端到端生产 SOP

### 6.1 总体闭环

```text
[内容入库]
    ↓
[诗意解析] → [人工校对/锁定历史约束]
    ↓
[生成 3 个画面方案]
    ↓
[风格匹配：偏好 + 探索]
    ↓
[Prompt 编译与版本固化]
    ↓
[预算/去重/并发检查]
    ↓
[图像生成]
    ↓
[自动质检 + 相似度去重]
    ↓
[今日候选池]
    ↓
[收藏 / 淘汰 / 标签 / 继续衍生]
    ↓
[偏好更新 + 新一轮任务]
    ↓
[成品锁定、排版、导出、归档]
```

### 6.2 标准作业步骤

| 步骤 | 输入 | 系统动作 | 输出 | 失败处理 |
|---|---|---|---|---|
| 1. 诗词入库 | 正文、作者、注释、来源 | 标准化、去重、来源登记 | `Poem` | 缺来源或正文冲突则待校对 |
| 2. 诗意解析 | `Poem` | 提取主题、情绪、意象、时空和不确定点 | `PoemAnalysis` | Schema 校验失败重试 1 次 |
| 3. 画面策划 | 解析结果、历史约束 | 生成叙事、意境、象征等不同方案 | `VisualPlan[]` | 相似方案合并或重生成 |
| 4. 风格分配 | 画面方案、风格包、偏好 | 分配稳定风格和探索风格 | `StyleAssignment` | 不兼容风格降权 |
| 5. Prompt 编译 | 诗、方案、风格、规格 | 按模板拼装正向/约束提示与参数 | `PromptVariant` | 缺字段时阻断，不静默补全 |
| 6. 预算闸门 | 任务、预算账本、队列状态 | 检查额度、并发、单诗/单风格上限 | `GenerationTask` | 超预算进入 `BudgetBlocked` |
| 7. 图像生成 | 编译结果、图像后端 | 幂等提交、轮询或接收结果 | `GeneratedImage` | 指数退避，最多重试 2 次 |
| 8. 自动质检 | 图片、诗、方案、风格 | 技术、相关性、历史和构图评分 | `ReviewResult` | 质检不可用时进入人工待审 |
| 9. 候选组装 | 通过图、评分、相似度 | 硬过滤、排序、去重、保证多样性 | `DailyCandidateSet` | 候选不足时补任务而非降门槛 |
| 10. 人工反馈 | 候选图 | 收藏、淘汰、优点标签、衍生意图 | `UserFeedback` | 无反馈不推断为不喜欢 |
| 11. 偏好更新 | 显式反馈、历史统计 | 更新风格/构图/配色权重 | `PreferenceProfile` | 保留至少 20% 探索配额 |
| 12. 成品归档 | 精选图、排版模板 | 锁定版本、合成文字、导出与记录 | `ExportAsset` | 成品修改产生新版本 |

### 6.3 每日运行节奏（建议）

| 时段 | 任务 | 建议配额 |
|---|---|---|
| 00:00-05:30 | 新诗与新风格探索 | 当日生成额度的 45% |
| 05:30-07:00 | 自动质检、去重、候选重排 | 不生成或仅补缺 |
| 07:00 | 生成“今日候选” | 20-40 张 |
| 08:00-18:00 | 用户筛选后触发精选衍生 | 当日额度的 35% |
| 18:00-23:00 | 修复任务与低覆盖诗补量 | 当日额度的 20% |

上述时段是默认策略，不应写死。用户可按电价、本地设备空闲时段、云端限流和个人作息调整。

### 6.4 人工每日 SOP

1. 查看“今日摘要”：总生成数、质检通过数、候选数、预算消耗和异常；
2. 先浏览系统推荐前 20，快速做“收藏 / 淘汰 / 稍后看”；
3. 对收藏图补充 1-3 个优点标签，如“留白、配色、少人物”；
4. 选择最多 5 张“继续衍生”，明确保持项和变化项；
5. 将已满意诗标记为“完成”或“锁定当前风格”；
6. 检查系统建议的下一轮配额，确认或调整后离开。

## 7. 功能架构

### 7.1 六大功能域

| 功能域 | 核心模块 | 主要职责 |
|---|---|---|
| 内容中心 | 诗词库、版本、来源、标签 | 管理结构化诗词和教育文本 |
| 创意中心 | 诗意解析、画面策划、风格包、人物设定 | 将文学内容转成视觉生产需求 |
| 生产中心 | Prompt 编译、任务队列、生成适配器、调度器 | 创建并执行批量生成任务 |
| 审核中心 | 自动质检、去重、候选池、人工反馈 | 降低废图并收集审美信号 |
| 资产中心 | 精选库、排版模板、导出、版权与来源记录 | 形成可交付资产 |
| 运营中心 | 预算、统计、告警、系统设置、审计日志 | 保证连续运行和成本可控 |

### 7.2 核心服务模块

```text
PoemRepository        诗词与来源管理
PoemAnalyzer          诗意结构化
VisualPlanGenerator   多方案画面策划
StylePackManager      风格包与版本管理
PromptCompiler        提示词编译与适配
GenerationScheduler   排产、预算和优先级
ImageGenerationWorker 图像后端调用
AutoReviewer          自动质检和打分
CandidateRanker       候选排序与多样性控制
FeedbackManager       人工反馈记录
IterationEngine       衍生与偏好驱动任务
ExportComposer        中文排版与多规格导出
```


### 8.2 关键对象关系

一首诗可以有多个分析版本；一个分析版本可以生成多个画面方案；每个画面方案可以匹配多个风格版本；每个“方案 × 风格 × 输出规格”形成一个 Prompt 变体；一个 Prompt 变体可产生多个生成任务与图片；图片拥有质检、反馈、衍生和导出记录。

## 9. 页面与交互设计

### 9.1 总览页

**目标：** 用户用 30 秒判断系统是否正常、今天是否值得筛选、预算是否安全。

主要区域：

- 今日生产：已生成、通过质检、候选、精选、失败；
- 预算进度：今日/本周消耗、预测收盘值、是否触发软限制；
- 今日推荐预览：前 8 张候选；
- 诗词完成度：未开始、探索中、已稳定、已完成；
- 风格表现：近 7 日收藏率与单位可用图成本；
- 异常卡片：生成器离线、任务积压、重复率过高、连续低分风格。

关键交互：点击任一指标进入已筛选列表；预算达到 70% 时顶部出现黄色提示，90% 时出现红色硬停止状态。

### 9.2 诗词列表页

字段：诗名、作者、主题、完成度、当前主风格、候选数、精选数、最近生成时间、状态。

筛选：作者、题材、季节、情绪、状态、风格、是否缺少插图。支持批量“加入今晚生产”“暂停探索”“标记完成”。

### 9.3 诗词详情页

采用五个页签：

1. **内容：** 正文、版本、注释、译文、来源；
2. **诗意解析：** 主题、情绪、意象、时空、历史约束、不确定点；
3. **画面方案：** 卡片式展示叙事/意境/象征方案，可锁定或编辑；
4. **生成记录：** 按方案、风格和批次查看所有图片；
5. **成品：** 精选图、排版模板与导出版本。

### 9.4 画面策划板

每个方案以“可编辑需求卡”呈现：标题、方案类型、主体、前中后景、镜头、动作、光线、配色、情绪、必须出现、禁止出现。支持：

- 一键生成 3 个差异化方案；
- 锁定某字段后重生成其他字段；
- 对比两个方案的差异；
- 标记“可生产 / 待修改 / 停用”；
- 查看该方案在不同风格下的历史命中率。

### 9.5 今日候选页

这是产品最高频页面，默认瀑布流 + 快捷键：

- `F` 收藏；
- `X` 淘汰；
- `I` 继续衍生；
- `C` 进入 A/B 对比；
- `1-5` 快速标记最喜欢的特征；
- 空格展开大图与质检详情。

单卡显示：诗名、风格、方案类型、综合分、异常标签、成本、衍生代数。默认隐藏完整 Prompt，避免干扰审美判断。

### 9.6 A/B 对比页

并排比较 2-4 张图，允许锁定比较维度：

- 同诗同风格，不同构图；
- 同诗同构图，不同风格；
- 原图与衍生图；
- 同风格跨诗一致性。

用户选择胜者后，可补充“胜在配色 / 构图 / 人物 / 意境 / 历史感”。系统记录成对偏好，价值高于单纯点赞。

### 9.7 风格实验室

包含风格包编辑器、基准诗选择、4-9 宫格小样、指标卡与版本对比。发布新版本必须经过至少 5 首基准诗的小样测试，并显示与旧版本的收藏率、偏题率和成本差异。

### 9.8 任务与队列页

按 `Queued / Running / Reviewing / Candidate / Failed / BudgetBlocked` 分组。支持暂停单风格、重试失败、降低并发、终止未开始批次。已运行任务不可直接删除，只能取消并保留审计记录。

### 9.9 成品导出页

提供以下模板：

- 4:5 竖版古典诗卡；
- 16:9 横版课程配图；
- 9:16 短视频画面；
- A4 / Letter 教学打印页；
- 儿童学习卡（正文、拼音、译文）；
- 纯插图与透明留白版。

文字合成由程序完成，支持宋体/楷体、字号、行距、署名、印章占位和安全边距预览。

## 10. 风格包体系

### 10.1 风格包定义

风格包不是一段自然语言，而是一份可版本化、可评估、可组合的美术规范。建议字段：

```json
{
  "style_id": "light_gongbi_v1",
  "name": "清雅淡彩工笔",
  "status": "active",
  "visual_traits": {
    "line": "细而克制的轮廓",
    "texture": "轻微熟宣纸纹理",
    "palette": ["月白", "黛青", "赭石", "浅绛"],
    "lighting": "柔和自然光",
    "contrast": "中低对比",
    "saturation": "低至中饱和",
    "whitespace": "中高"
  },
  "character_design": {
    "proportion": "写实偏绘本",
    "expression": "含蓄自然",
    "costume": "经审核的唐代服饰描述"
  },
  "suitable_for": ["思乡", "山水", "春景", "夜景"],
  "avoid": ["3D 游戏渲染", "大眼动漫", "现代器物", "画面文字"],
  "reference_asset_ids": [],
  "prompt_fragment_version": "1.0.0"
}
```

### 10.2 首批 6 个风格包

| 风格包 | 核心特征 | 适用题材 | 主要风险 |
|---|---|---|---|
| 清雅淡彩工笔 | 细线、低饱和、纸本质感 | 人物、庭院、思乡 | 容易过度精致、显得明清化 |
| 极简水墨留白 | 大留白、墨色层次、少人物 | 山水、边塞、禅意 | 诗意信息不足、画面过空 |
| 东方儿童绘本 | 柔和造型、清晰叙事、亲和配色 | 启蒙、低龄课程 | 容易 Q 版化或历史感不足 |
| 新中式装饰插画 | 平面构成、图案化、强版式 | 海报、卡片、节气 | 装饰压过诗意 |
| 温暖手绘自然系 | 手绘笔触、自然光、温暖叙事 | 田园、友情、春景 | 可能出现现代动画语汇 |
| 月夜青绿叙事 | 青绿与月白、冷暖对照 | 夜景、羁旅、思乡 | 夜景同质化、月亮符号滥用 |

风格命名应描述视觉特征，避免直接使用具体在世艺术家、知名动画工作室或受保护 IP 的名字。

### 10.3 风格包生命周期

```text
Draft → Benchmarking → Active → Limited → Retired
```

- `Draft`：仅编辑，不进入自动调度；
- `Benchmarking`：仅在基准诗上小规模测试；
- `Active`：可进入稳定生产；
- `Limited`：连续低命中或成本异常，限制配额；
- `Retired`：停止新任务，但保留历史可回放性。

### 10.4 风格一致性机制

- 固定色板、线条、材质、人物比例和常用构图；
- 为李白、杜甫等高频人物建立独立“角色设定包”；
- 同一成品系列锁定风格主版本，不自动跟随最新版本；
- 跨诗抽样进行一致性评估，而非只评价单张图；
- 参考图必须记录来源、使用授权和可使用范围。

## 11. 诗意解析系统

### 11.1 目标

将文学文本转成可供策划和生成系统消费的结构化语义，同时保留不确定性，避免模型将有争议的解释当作确定事实。

### 11.2 输出 Schema

```json
{
  "summary": "一句话叙事概括",
  "theme": ["思乡"],
  "core_emotion": "克制而深沉的乡愁",
  "emotion_curve": ["静谧", "触景", "思念"],
  "imagery": ["月光", "霜感", "旅舍", "远方"],
  "time": "夜晚",
  "season": "不确定",
  "location": ["旅舍室内"],
  "people": ["诗人"],
  "must_have": ["月光形成霜感"],
  "optional": ["床榻", "窗格", "远山"],
  "historical_constraints": ["唐代服饰与木构室内"],
  "forbidden_elements": ["现代家具", "电灯", "玻璃幕墙"],
  "ambiguities": [
    {"field": "床", "note": "具体物件解释存在讨论，不应过度写实化"}
  ],
  "confidence": 0.82,
  "source_version": "poem_text_v1"
}
```

### 11.3 解析规则

1. 先忠实提取诗中明确意象，再提出可视化推断；
2. 将“事实”“合理推断”“艺术化处理”分开记录；
3. 有争议的地点、器物和人物年龄进入 `ambiguities`；
4. 解析结果必须通过 JSON Schema 校验；
5. 同一诗的人工校订版本可锁定，自动任务不得覆盖；
6. 现代注释、译文和赏析必须记录版权来源，不直接抓取未授权内容。

### 11.4 人工校对入口

编辑可对历史约束、禁用元素和不确定点进行锁定。锁定字段在后续重分析时保留，并在界面中显示“人工确认”标识。

## 12. 画面策划系统

### 12.1 方案类型

每首诗默认从下列类型中选择 3 个差异足够大的方案：

1. **叙事型：** 呈现人物、动作和情节；
2. **意境型：** 弱化人物，突出环境、光线和留白；
3. **象征型：** 用物象或空间对照表达情绪；
4. **儿童绘本型：** 清晰、温和、适合学习理解；
5. **装饰海报型：** 强构成、预留文字区、适合传播。

### 12.2 方案 Schema

```json
{
  "plan_id": "plan_jingyesi_mood_01",
  "plan_type": "mood",
  "title": "月光入室",
  "visual_thesis": "以月光的霜感替代直白表情，表现思乡",
  "subject": "诗人背影，可弱化",
  "scene": "唐代旅舍室内",
  "foreground": ["床沿", "竹简"],
  "midground": ["诗人低头"],
  "background": ["木窗", "圆月", "薄雾远山"],
  "camera": "室内中远景，略侧后方",
  "composition": "人物右三分之一，窗月左侧，大量暗部留白",
  "lighting": "清冷月光与微弱暖灯",
  "palette_intent": "月白、黛青、少量赭石",
  "must_have": ["地面霜感光影"],
  "avoid": ["夸张悲伤表情", "仙侠光效"],
  "text_safe_area": "左上区域"
}
```

### 12.3 差异性规则

- 三个方案的主视觉中心不得完全相同；
- 至少一个方案减少人物存在感；
- 至少一个方案提供明确文字安全区；
- 同一批次不得只改变形容词而复用相同构图；
- 新方案与历史方案进行文本与结构相似度检查，超过阈值则重新策划。

### 12.4 方案评分

策划阶段先不出图，使用规则或模型评估：诗意覆盖、视觉清晰度、历史可行性、风格适配、差异度和排版可用性。低于最低分的方案不进入生产队列。

## 13. Prompt 编译系统

### 13.1 定位

Prompt Compiler 是产品核心中间层。它将诗意解析、画面方案、风格包、角色设定、输出规格与生成器能力编译成稳定的机器输入，屏蔽不同模型的参数差异。

### 13.2 编译顺序

```text
视觉命题
→ 主体与动作
→ 场景与前中后景
→ 构图与镜头
→ 光线与配色
→ 风格包片段
→ 历史约束
→ 排版安全区
→ 质量要求
→ 禁止元素
→ 输出尺寸与生成器参数
```

### 13.3 输出结构

```json
{
  "compiler_version": "2.1.0",
  "provider_adapter": "local_comfy_v1",
  "poem_id": "tang_001",
  "analysis_version": 3,
  "plan_id": "plan_007",
  "style_version_id": "style_light_gongbi_1_2",
  "positive_prompt": "...",
  "negative_prompt": "...",
  "aspect_ratio": "4:5",
  "width": 1024,
  "height": 1280,
  "seed_policy": "random",
  "reference_asset_ids": [],
  "prompt_hash": "sha256:..."
}
```

### 13.4 关键规则

- 不将诗句作为画面内文字生成；
- 禁止项按“业务禁用、历史禁用、技术缺陷、风格偏离”分组；
- 使用稳定字段顺序，方便版本对比和回归测试；
- 同一输入产生相同 `prompt_hash`，避免重复任务；
- 风格包只提供可复用片段，不覆盖诗意和构图的核心要求；
- 生成器不支持的参数由适配器降级，并记录降级原因；
- Prompt 模板变更必须对基准集做回归，不直接全量发布。

### 13.5 Prompt 版本策略

采用语义版本：

- 主版本：输出结构或语义策略不兼容；
- 次版本：新增字段、改变编译片段或默认权重；
- 修订版本：错别字、轻微约束修正。

成品图始终绑定当时的完整编译结果，不能只保存模板引用。

## 14. 24 小时自动调度系统

### 14.1 任务类型

| 类型 | 目的 | 默认优先级 |
|---|---|---:|
| `Iteration` | 基于用户明确选中的图继续衍生 | 100 |
| `Repair` | 修复可定位问题，如留白不足、人物过多 | 85 |
| `Coverage` | 为缺少可用图的诗补齐基础覆盖 | 70 |
| `Exploit` | 扩大高命中风格和构图 | 60 |
| `Explore` | 尝试新风格、新镜头和新组合 | 40 |
| `Benchmark` | 风格包或 Prompt 版本基准测试 | 30 |

### 14.2 优先级公式

建议使用可解释的评分，而不是直接让模型决定：

```text
Priority =
  0.30 × 诗词未完成度
+ 0.25 × 人工反馈强度
+ 0.15 × 风格历史命中率
+ 0.15 × 新颖度需求
+ 0.15 × 等待时长
- 成本惩罚
- 重复风险
```

权重由运营配置，系统保留最终分数和组成项，方便排查“为什么生成了这张”。

### 14.3 探索与利用配比

MVP 建议：

- 50%：基于已验证风格和用户偏好扩展；
- 30%：新构图、新配色、新风格探索；
- 20%：修复、补覆盖与基准测试。

当某首诗连续出现 3 张精选图后，自动降低探索频率；当某风格连续 20 次低命中时，进入 `Limited` 并暂停 24 小时。

### 14.4 调度安全机制

- 全局每日张数上限；
- 每首诗、每风格、每方案的日上限；
- 同源图片最多衍生 2-3 代；
- 幂等任务键，防止服务重启后重复提交；
- Worker 租约和心跳，防止“运行中”任务永久卡住；
- 指数退避与抖动，最多自动重试 2 次；
- 预算软阈值、硬阈值和紧急停止；
- 生成器健康检查与自动熔断；
- 队列公平性，避免热门诗占满全部算力。

### 14.5 触发规则示例

- 新诗入库：创建 3 个方案 × 3 个风格的低成本首轮任务；
- 用户收藏：创建 4 个变体，锁定风格和视觉命题；
- 用户标记“构图喜欢”：保持构图，变化光线与配色；
- 用户标记“不要人物”：后续同诗 48 小时内降低人物方案权重；
- 连续低分：暂停组合并生成问题报告，不盲目重试；
- 候选池不足：补高置信任务，不降低质检硬门槛。

## 15. 候选池与人工筛选

### 15.1 候选生成流程

```text
生成图片
→ 安全与技术硬过滤
→ 诗意/风格/历史评分
→ 感知哈希与语义去重
→ 综合排序
→ 多样性重排
→ 按诗和风格设置展示配额
→ 形成今日候选
```

### 15.2 硬过滤

以下情况不进入候选池：

- 违反安全或平台政策；
- 文件损坏、尺寸错误或下载失败；
- 明显乱码文字、水印或大面积 Logo；
- 与原诗几乎无关；
- 明显现代物件且无法解释；
- 与同批高位图片高度重复；
- 已在历史中被用户明确标记为“不要再生成这类”。

### 15.3 排序与多样性

建议综合分：

```text
CandidateScore =
  0.30 × 诗意相关
+ 0.20 × 风格匹配
+ 0.15 × 构图质量
+ 0.10 × 历史合理
+ 0.10 × 技术完整
+ 0.10 × 用户偏好
+ 0.05 × 新颖度
```

排序后执行多样性重排：同诗同风格连续最多 2 张，同一源图的变体最多占候选池 15%，探索图至少占 20%。

### 15.4 候选分层

- **重点推荐：** 综合分高且与历史偏好匹配；
- **值得探索：** 新颖度高，但偏好置信度不足；
- **可修复：** 方向正确但有明确局部问题；
- **人工复核：** 模型评分冲突或历史争议明显。

## 16. 人工反馈与偏好学习

### 16.1 反馈层级

#### 一级：快速行为

- 收藏；
- 淘汰；
- 稍后看；
- 继续衍生；
- 标记成品。

#### 二级：喜欢什么

- 风格；
- 构图；
- 配色；
- 光线；
- 意境；
- 人物设计；
- 历史感；
- 留白。

#### 三级：需要怎么改

- 减少人物；
- 增加留白；
- 降低饱和；
- 保持构图换光线；
- 保持风格换镜头；
- 修正器物/服饰；
- 去除文字；
- 更适合儿童。

### 16.2 MVP 偏好模型

第一版不训练复杂模型，采用可解释权重：

```text
PreferenceProfile = {
  style_weights,
  composition_weights,
  palette_weights,
  character_density_preference,
  whitespace_preference,
  poem_category_style_matrix,
  negative_constraints
}
```

每次明确收藏增加权重，淘汰降低权重，成对比较的胜者信号权重高于普通收藏。无操作、停留时间短或未曝光图片不作为负反馈。

### 16.3 防止审美收窄

- 永久保留至少 20% 探索配额；
- 偏好按“全局、题材、单诗”分层，不把夜景偏好强行应用到所有诗；
- 对高相似候选降权；
- 每周展示少量“超出偏好但质量高”的惊喜候选；
- 用户可一键重置某项偏好或进入“纯探索模式”。

### 16.4 反馈到任务的映射

| 用户反馈 | 任务变化 |
|---|---|
| 收藏 + 风格喜欢 | 该风格在同题材权重上调 |
| 构图喜欢 | 锁定构图，生成光线/配色变体 |
| 不要人物 | 降低人物方案权重并加入负向约束 |
| 继续衍生 | 创建高优先级 `Iteration` 任务 |
| 标记成品 | 诗词完成度上升，停止常规探索 |
| 淘汰 + 历史错误 | 记录缺陷标签，触发规则修复 |

## 17. 自动质检系统

### 17.1 质检维度

| 维度 | 检查内容 | 是否可硬拒绝 |
|---|---|---|
| 安全合规 | 不当内容、未成年人风险、平台政策 | 是 |
| 技术完整 | 文件损坏、分辨率、裁切、透明度 | 是 |
| 诗意相关 | 是否表达核心意象和情绪 | 低于极限时是 |
| 风格匹配 | 是否符合风格包线条、色彩、材质 | 否 |
| 历史合理 | 服饰、器物、建筑、时代穿帮 | 严重时是 |
| 构图质量 | 主次、留白、视线、文字安全区 | 否 |
| 人物质量 | 手、脸、姿态、人数 | 严重时是 |
| 文本噪声 | 乱码、字母、水印、Logo | 是 |
| 系列一致 | 与目标系列在角色和风格上是否一致 | 否 |

### 17.2 输出 Schema

```json
{
  "overall_score": 82,
  "poem_relevance": 88,
  "style_match": 84,
  "historical_plausibility": 72,
  "composition": 86,
  "technical_integrity": 95,
  "series_consistency": 78,
  "has_unexpected_text": false,
  "hard_fail_reasons": [],
  "problems": [
    {"code": "WINDOW_STYLE_UNCERTAIN", "severity": "medium", "note": "窗格细节偏晚期"}
  ],
  "decision": "candidate",
  "reviewer_version": "vision_review_v1.3"
}
```

### 17.3 建议门槛

- 安全、文件完整性、文字噪声：硬门槛；
- 综合分 `< 60`：自动淘汰；
- `60-74`：可修复或人工复核；
- `75-84`：进入普通候选；
- `>= 85`：进入重点推荐；
- 历史合理性低于 60 时，不得作为教育成品直接导出。

门槛必须可配置，并通过人工抽检校准。模型评分是排序信号，不应被当作绝对真值。

### 17.4 质检闭环

每周从“自动淘汰、普通候选、重点推荐”各抽样 20 张人工复核，计算误杀率和漏放率。若某项评分与人工判断持续偏离，回滚质检版本或调整阈值。

## 18. 任务状态机

### 18.1 生成任务状态

```text
Draft
  ↓
Queued ↔ BudgetBlocked
  ↓
Running
  ├─→ RetryWaiting → Queued
  ├─→ Failed
  └─→ Generated
          ↓
      Reviewing
       ├─→ AutoRejected
       ├─→ HumanReview
       └─→ Candidate
                ↓
        Selected / Rejected / Iterating / Archived
                ↓
             Exported
```

### 18.2 状态定义

| 状态 | 含义 | 允许动作 |
|---|---|---|
| `Draft` | 参数尚未完成 | 编辑、删除草稿 |
| `Queued` | 已通过预算和去重检查 | 暂停、取消 |
| `BudgetBlocked` | 因预算或配额阻断 | 等待重置、人工放行 |
| `Running` | Worker 已领取 | 观察、超时回收 |
| `RetryWaiting` | 等待下一次退避 | 取消、提前重试 |
| `Generated` | 图像已保存 | 进入质检 |
| `Reviewing` | 正在自动质检 | 等待、超时降级 |
| `AutoRejected` | 命中硬过滤 | 人工申诉、归档 |
| `HumanReview` | 自动判断不确定 | 人工决定 |
| `Candidate` | 可进入候选池 | 收藏、淘汰、衍生 |
| `Selected` | 人工精选 | 排版、导出、衍生 |
| `Iterating` | 已产生后续变体 | 查看谱系 |
| `Archived` | 不再活跃 | 恢复、查询 |
| `Failed` | 重试耗尽 | 查看错误、手动重试 |
| `Exported` | 已产生交付资产 | 创建新导出版本 |

### 18.3 状态机要求

- 所有转换由服务端校验，前端不能直接写任意状态；
- 每次状态转换记录操作人、原因、时间和关联任务；
- `Running` 超过租约自动回到 `Queued` 或 `Failed`；
- 已导出成品的源图不可物理删除，只能进入受限归档；
- 重试创建新的尝试记录，不覆盖原始错误。

### 19.2 数据约束

- `prompt_hash + provider + generation_parameters_hash` 建唯一约束，防止重复；
- `style_versions` 与 `poem_analyses` 只新增版本，不覆盖历史；
- 图片文件进入对象存储，数据库只存对象键、校验和和元数据；
- `user_feedback` 保留撤销记录，不做原地删除；
- 成本同时记录估算与实际值，允许对账；
- `generated_images` 保存谱系根、父图和衍生深度；
- 对状态、日期、诗词、风格和 `prompt_hash` 建索引。

### 19.3 数据保留

- 成品与其完整生产链长期保留；
- 自动淘汰图可按 30-90 天策略转冷存储或删除；
- 任务日志保留至少 180 天；
- 用户删除请求应覆盖原图、缩略图、导出物和缓存；
- 删除前检查是否被成品、项目或授权审计引用。

## 21. Codex 无 API 运行方案

### 21.1 可行性结论

没有 OpenAI API 时，可以用 Codex 搭建和执行一套“低频、分批、可恢复”的生产原型。Codex 适合担当生产经理、策划师和脚本执行器，但不是无限量图片生成服务。真正出图仍需以下任一后端：

1. 本地 ComfyUI 或其他本地图像工作流；
2. 人工将 Codex 生成的任务和 Prompt 提交到 ChatGPT；
3. 合法且允许自动化的第三方生成服务；
4. 后续接入正式图像 API。

### 21.2 推荐项目结构

```text
tang-poem-studio/
├─ AGENTS.md
├─ skills/tang-poem-production/SKILL.md
├─ data/poems.json
├─ styles/*.json
├─ state/generation.db
├─ queue/pending/
├─ queue/running/
├─ scripts/select_jobs.py
├─ scripts/compile_prompts.py
├─ scripts/generate_images.py
├─ scripts/review_images.py
├─ scripts/build_gallery.py
├─ outputs/YYYY-MM-DD/
└─ logs/
```

### 21.3 Codex Scheduled Task 模式

建议每 1-2 小时运行一个短批次，而不是让单次任务永久占用：

```text
读取 SKILL.md 与状态数据库
→ 恢复超时任务
→ 检查今日预算和生成上限
→ 选择下一批 5-10 个任务
→ 运行编译与本地生成脚本
→ 更新状态与成本
→ 生成候选画廊和摘要
→ 正常退出
```

当前官方文档说明，项目范围的 Scheduled Task 在后台运行时需要机器保持开机、桌面应用运行，且项目仍可从磁盘访问；任务以默认沙箱权限无人值守执行。因此，原型应使用最小权限、可重复脚本和严格的路径范围。

### 21.4 `codex exec` 模式

也可由系统计划任务调用非交互命令：

```bash
codex exec --cd /path/to/tang-poem-studio \
  "读取生产 skill，执行一轮不超过 10 张的生成批次；更新状态并输出摘要。"
```

官方文档将 `codex exec` 定位为适合脚本或 CI 风格、无需人工交互并在完成后退出的运行方式。实际部署时应使用正常沙箱和明确目录权限，不使用跳过审批与沙箱的危险参数。

### 21.5 三阶段无 API 路线

| 阶段 | 方案 | 自动化程度 | 主要限制 |
|---|---|---:|---|
| A | Codex 产出策划与 Prompt，人工在 ChatGPT 出图 | 40% | 仍需人工提交和保存 |
| B | Codex + 本地 ComfyUI + SQLite + HTML 画廊 | 80% | 依赖本机算力、模型许可证和稳定运行 |
| C | Codex 仅负责编排与维护，生产迁移至服务端 | 90%+ | 需要正式 API、队列和运维 |

### 21.6 无 API 方案的边界

- ChatGPT/Codex 套餐不应被设计成无限量、无成本的 API 替代；
- 网页点击自动化脆弱，可能受登录、验证码、页面变化和服务条款影响；
- Mac 本地图像生成速度与可用模型取决于硬件；
- Scheduled Task 适合分批工作和报告，不替代数据库队列的精细调度；
- 生产状态必须保存在文件或数据库中，不能只依赖对话上下文。

## 22. ChatGPT / OpenAI API 未来接入

### 22.1 接入原则

未来接入不改变业务状态机，只替换或新增智能节点和生成器适配器：

| SOP 节点 | 建议能力 | 系统产物 |
|---|---|---|
| 诗意解析 | Responses API + Structured Outputs | `PoemAnalysis` |
| 画面策划 | Responses API + Structured Outputs | `VisualPlan[]` |
| 风格匹配 | 文本模型 + 规则 | `StyleAssignment[]` |
| Prompt 编译 | 文本模型辅助 + 确定性编译器 | `PromptVariant` |
| 插图生成 | Image API 或 Responses 图像工具 | `GeneratedImage` |
| 图片质检 | 视觉输入 + Structured Outputs | `ReviewResult` |
| 偏好总结 | 文本模型 + 统计 | `PreferenceSummary` |
| 调度、预算、重试 | 自有代码 | `GenerationTask` / `BudgetLedger` |

OpenAI 官方文档将 Responses API 推荐为新项目的统一接口，并支持多轮、工具和多模态流程；Structured Outputs 可使文本输出符合指定 JSON Schema，适合本产品的解析、策划和质检中间结果。

### 22.2 图像接口选择

- 单次独立生成或编辑：使用 Image API，流程简单、模型选择和参数更直接；
- 多轮对话式迭代、参考图持续编辑：使用 Responses API 的图像生成工具；
- 批量生产系统仍由自有 Scheduler 控制，模型不决定预算和任务状态；
- 产品代码通过“能力注册表”选择支持尺寸、编辑、参考图、透明背景等能力的模型，不硬编码营销名称。

### 22.3 ChatGPT 控制入口

当后端 API 稳定后，可将系统能力通过受控工具或 MCP/App 形式暴露给 ChatGPT，让用户用自然语言完成：

- “今晚优先生成李白的 5 首诗，总量不超过 80 张”；
- “降低儿童绘本风权重，保留 25% 探索”；
- “把今天收藏的 6 张都生成少人物版本”；
- “总结过去 7 天哪种风格成本最低、收藏率最高”。

ChatGPT 是控制面和解释层，不是 24 小时执行引擎；后端仍负责权限、状态、幂等、预算、审计和实际任务执行。

### 22.4 安全要求

- API Key 不进入前端、移动端包或 Prompt；
- 工具调用使用最小权限和参数白名单；
- 写操作区分预览、确认和执行；
- 高成本批次、批量删除和对外发布必须二次确认；
- 记录模型、请求版本、响应 ID、用量和失败原因；
- 不在 Prompt 中携带无必要的个人数据或未授权内部资料。

## 23. 成本与预算控制

### 23.1 成本模型

```text
日总成本 =
  文本解析与策划成本
+ 图像生成成本
+ 视觉质检成本
+ 存储与流量成本
+ 本地算力或云主机成本
+ 失败与重试成本
```

具体单价随模型和服务变化，产品内应读取配置或账单数据，不将价格写死在业务逻辑中。

### 23.2 关键成本指标

- 每生成图成本；
- 每候选图成本；
- 每收藏图成本；
- 每最终成品成本；
- 失败与自动淘汰浪费率；
- 单诗达到“完成”状态的累计成本；
- 各风格的收藏率 / 成本比。

### 23.3 预算闸门

| 阈值 | 动作 |
|---|---|
| 50% | 正常运行，展示趋势 |
| 70% | 软提醒，减少低优先级探索 |
| 85% | 仅执行精选衍生和修复任务 |
| 90% | 停止创建新生成任务 |
| 100% | 硬停止，仅允许质检、归档和报告 |

### 23.4 配额策略

- 日、周、月三级预算；
- 全局、项目、诗词、风格和用户五级配额；
- 高质量模式仅用于已验证方向；
- 首轮探索用较低分辨率或较低成本配置；
- 候选确定后再做高质量编辑或放大；
- 连续失败和连续低分自动熔断；
- 去重在生成前后各执行一次；
- 报表同时显示“预计花费”和“实际花费”。

### 23.5 容量估算模板

```text
每日图片数 = 诗词数 × 每诗方案数 × 每方案风格数 × 每组合张数

例：
10 首 × 2 方案 × 2 风格 × 2 张 = 80 张/日

每日预计成本 =
80 × 单张图片成本
+ 文本调用次数 × 单次文本成本
+ 80 × 单次质检成本
+ 存储与流量
```

系统应在用户保存批次前实时展示预计数量、预计成本和预算占比。

## 24. MVP 范围与验收标准

### 24.1 MVP 必做

- 30 首诗词数据与来源字段；
- 结构化诗意解析与人工锁定；
- 每首诗 3 个画面方案；
- 6 个风格包及版本管理；
- Prompt 编译器与去重哈希；
- 一个图像生成适配器（本地或正式 API）；
- 批次、队列、重试、暂停、预算上限；
- 自动技术检查 + 基础视觉质检；
- 今日候选、收藏、淘汰、标签、继续衍生；
- 简单偏好权重与 20% 探索保留；
- 精选库与纯图 / 诗卡导出；
- 日报、成本统计和错误日志。

### 24.2 MVP 暂不做

- 唐诗三百首全量录入；
- 复杂多人权限、审批流和客户空间；
- 训练专属美术模型、LoRA 管理平台；
- 自动发布到社交媒体；
- 风格市场和交易；
- 移动端原生 App；
- 复杂机器学习推荐；
- 全自动历史事实认证；
- 多语言和多地区商业化。

### 24.3 MVP 验收标准

| 维度 | 验收标准 |
|---|---|
| 闭环 | 可从诗词入库走到候选、反馈、衍生和导出 |
| 无人值守 | 连续 7 天按计划运行，无任务永久卡死 |
| 恢复能力 | 服务重启后未完成任务可恢复且不重复计费 |
| 候选效率 | 用户 20 分钟内可完成 20-40 张候选筛选 |
| 质量 | 质检硬过滤对乱码、损坏和明显偏题有效 |
| 偏好 | 连续反馈后，高偏好风格分配比例可观察地变化 |
| 成本 | 日预算硬上限有效，超限不再创建生成任务 |
| 追溯 | 任一成品可追溯到诗、方案、风格、Prompt 和任务 |
| 导出 | 至少可导出纯图与一种带中文排版的诗卡 |

## 25. 开发里程碑

### 25.1 建议路线

| 里程碑 | 周期 | 主要交付 | 退出条件 |
|---|---:|---|---|
| M0 产品验证 | 1 周 | 10 首诗、3 风格、手工跑通任务样例 | 能选出至少 5 张方向明确的候选 |
| M1 数据与策划 | 2 周 | 诗词库、解析 Schema、方案编辑器、风格包 | 中间产物可版本化和回放 |
| M2 生成流水线 | 2 周 | Prompt 编译、适配器、队列、状态机、预算 | 可稳定跑完一个 50 张批次 |
| M3 候选与反馈 | 2 周 | 质检、去重、候选页、快捷键、反馈 | 每日筛选闭环可用 |
| M4 自动迭代 | 1-2 周 | 调度策略、偏好权重、衍生谱系、日报 | 连续运行 7 天 |
| M5 导出与硬化 | 1-2 周 | 诗卡、监控、备份、合规字段、验收 | MVP 发布候选 |

### 25.2 团队配置

最小团队可由 1 名全栈/后端工程师 + 1 名产品/内容编辑兼任；若追求更快和更高质量，增加前端设计工程师与历史内容顾问。个人业余开发时，应优先做本地单用户版，避免过早建设账号、组织和复杂部署。

### 25.3 每阶段评审材料

- 可演示的工作流；
- 基准诗和基准图集；
- 生成数量、收藏率、失败率、重复率和成本；
- 关键问题清单与下一阶段范围；
- 风格包和 Prompt 版本变化；
- 风险与合规检查结果。

## 26. 风险与合规

### 26.1 内容版权

- 唐诗原文本身通常属于公版内容，但现代整理、注释、译文、赏析、拼音标注和版式可能受版权保护；
- 诗词数据应记录来源、版本和许可，优先使用权威公版资料、自有整理或明确授权内容；
- 不直接复制网络文章中的现代译文和赏析；
- 参考图、纹理、字体和本地模型都要记录许可证及商业使用范围。

### 26.2 风格与知识产权

- 风格包使用可描述的视觉特征，不以模仿具体在世艺术家或知名 IP 为卖点；
- 禁止生成明显商标、角色、品牌包装或未经授权的 Logo；
- 若用户上传参考图，要求确认拥有使用权；
- 对外商业交付保留生成记录、人工修改记录和素材来源。

### 26.3 历史与文化风险

- 自动历史评分只表示“合理性”，不表示学术认证；
- 对有争议的器物、场景和人物形象明确标记不确定性；
- 教育出版成品需人工编辑或顾问复核；
- 避免将文学意象误包装为确定历史事实。

### 26.4 AI 内容与未成年人

- 面向儿童的内容需更严格的安全过滤和年龄适宜性规则；
- 不生成恐怖、血腥、性化或不当人物表现；
- 对 AI 生成内容提供适当标识与可追溯信息；
- 若收集学生数据，遵循适用的隐私与未成年人保护要求。

### 26.5 技术与运营风险

| 风险 | 后果 | 缓解措施 |
|---|---|---|
| 成本失控 | 夜间大量消费 | 多级预算、估算、硬停止、熔断 |
| 任务重复 | 重复计费与污染候选 | 幂等键、Prompt 哈希、租约 |
| 风格漂移 | 整套作品不一致 | 风格版本锁定、跨诗基准测试 |
| 自动质检误判 | 好图被淘汰或坏图进入候选 | 分层候选、人工抽检、可申诉 |
| 网页自动化失效 | 无人值守中断 | 不作为生产主链路 |
| 本地设备离线 | 夜间任务未执行 | 健康检查、补跑、未来迁移服务端 |
| 模型或接口变化 | 生产中断 | 提供商适配层、能力注册表、回归测试 |
| 数据丢失 | 成品与反馈不可恢复 | 数据库备份、对象校验、异地备份 |

### 26.6 安全基线

- 最小权限沙箱；
- 密钥不入库明文、不入日志、不入 Prompt；
- 管理操作审计；
- 对外 URL 使用短期签名；
- 上传文件做类型、尺寸和恶意内容检查；
- 定期备份与恢复演练；
- 批量删除、预算放行和对外发布需显式确认。

## 27. 商业模式

### 27.1 商业化路径

#### 个人订阅

提供诗词库、每日配额、风格包、候选筛选和基础导出。适合教师、家长、学生和个人创作者。

#### 创作者专业版

提供批量生产、更多风格、自定义风格包、参考图、高清导出、品牌模板和高级统计。

#### 教育 / 出版团队版

提供团队空间、权限、项目审批、授权记录、批量导出、审计日志和私有部署选项。

#### 项目制服务

为教材、绘本、博物馆或文化机构提供整套风格策划、试稿、生成与人工精修交付。

#### 平台与生态

未来可提供内容包、模板包、合规风格包、开发 API 或 SDK，但必须先解决素材权利、质量标准和责任边界。

### 27.2 定价逻辑

定价不只按生成张数，还应体现：并发、风格资产、团队协作、高清导出、版权与审计、私有化和人工服务。对高成本能力采用额度或加购，避免“无限生成”带来的不可控风险。

### 27.3 可持续壁垒

- 诗词到视觉的结构化数据集；
- 经人工校准的历史约束和错误库；
- 风格包、角色设定与跨诗基准集；
- 用户成对偏好和衍生谱系；
- Prompt 编译器与多模型适配；
- 自动质检和单位可用图成本优化能力；
- 教育模板、工作流和版权审计能力。

## 28. 关键指标

### 28.1 北极星指标

**每周进入成品库的人工确认可用插图数。**

它同时反映系统是否持续运行、候选是否有价值、用户是否愿意筛选，以及最终是否形成可交付资产。

### 28.2 漏斗指标

```text
任务创建数
→ 生成成功数
→ 自动质检通过数
→ 候选曝光数
→ 人工收藏数
→ 继续衍生数
→ 成品锁定数
→ 导出使用数
```

### 28.3 核心指标定义

| 指标 | 定义 | MVP 参考目标 |
|---|---|---:|
| 生成成功率 | 成功落盘图片 / 已执行任务 | ≥ 90% |
| 候选率 | 进入候选池 / 成功图片 | 建立基线，不盲目追高 |
| 收藏率 | 收藏 / 已看候选 | ≥ 15% 为初期观察线 |
| 成品转化率 | 成品锁定 / 收藏 | 持续提升 |
| 单位可用图成本 | 总成本 / 成品锁定数 | 周环比下降 |
| 筛选效率 | 完成一日候选所需时间 | ≤ 20 分钟 |
| 重复率 | 高相似图片 / 成功图片 | ≤ 10% |
| 质检误杀率 | 人工判好但被自动淘汰 | < 5% 的长期目标 |
| 诗词覆盖率 | 至少 1 张成品的诗 / 目标诗数 | 30 首在 4 周内 ≥ 80% |
| 连续运行稳定性 | 无人工修复完成的计划批次 | 连续 7 天 |

### 28.4 诊断指标

- 各诗、各风格、各方案类型的收藏率；
- 不同衍生代数的边际收益；
- 自动质检分与人工判断相关性；
- 单风格连续低分次数；
- 队列等待时长、Worker 利用率和超时率；
- 预算预测误差；
- 用户反馈标签覆盖率；
- 已完成诗重新打开的比例。

### 28.5 指标使用原则

不以“生成更多”为增长目标。若图片总量上升但收藏率、成品数和单位成本变差，应降低产量并优先修复策划、风格与质检。

## 结论

《唐诗三百首 AI 插图生成器》的合理形态不是单次生成工具，而是一座小型、可控、持续学习的内容工厂。第一版最值得投入的不是复杂推荐算法，而是以下五项基础能力：

1. 结构化诗意解析；
2. 可版本化的画面方案与风格包；
3. 可替换的 Prompt 编译和生成适配层；
4. 受预算与状态机控制的持续调度；
5. 高效率候选筛选与明确反馈闭环。

只要这五项打通，即使初期采用 Codex + 本地脚本 + 人工出图，未来也能平滑迁移到本地 ComfyUI、OpenAI API 或其他图像服务，而不需要推翻产品数据、SOP 和用户反馈资产。
